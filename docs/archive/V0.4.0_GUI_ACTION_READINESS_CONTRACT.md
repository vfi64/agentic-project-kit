# v0.4.0 GUI Action Readiness Contract

Status: proposed
Decision status: proposed
Lifecycle: active

## Purpose

This contract defines the minimum action-readiness rules before the v0.4.0 GUI MVP may execute actions. It is a contract and test slice only. It does not implement GUI windows, widgets, buttons, release actions, or destructive workflows.

## Required action metadata

Every GUI-visible action must provide:

- a stable action name;
- a user-facing description;
- a safety class;
- a minimum Access level for visibility (`basic`, `advanced`, or `maintainer`);
- parameter rules or an explicit no-parameter declaration;
- expected outcomes;
- evidence behavior;
- execution-mode requirements.

## MVP safety classes

- Read-only actions may be visible and executable in the first GUI MVP.
- Bounded local actions may be visible only when their evidence and mode behavior is explicit.
- Destructive, release, publish, merge, tag, delete, and remote-mutating actions must be disabled in the first GUI MVP.
- Disabled actions may be shown for orientation, but they must not be executable from the first GUI MVP.

## Access-level visibility

Access level is a visibility convenience, not a permission boundary. The Tkinter cockpit may filter the visible action inventory by `basic`, `advanced`, or `maintainer`, but this must not change action executability. Safety classes and the existing action layer remain the execution boundary.

## Execution boundary

The GUI must use the existing Python action registry or action specs. It must not create a second command registry. It must not call removed shell adapters. It must not bypass the mode guard, typed execution contract, summary renderer, or evidence discipline.

## Evidence boundary

A GUI action result must expose whether evidence is not required, local only, or remote-backed. A GUI action must not show success when the underlying command failed, when the working tree is unexpectedly dirty, or when a required terminal log is missing.

## First GUI MVP allowed surface

The first GUI MVP may start with status, action inventory, read-only diagnostics, doctor, check-docs, mode-check, and other safe inspection routes. It must not execute release-publish, merge, tag, delete, DOI write, or remote mutation routes.
