# v0.4.0 GUI MVP Roadmap

Status: proposed
Decision status: proposed
Lifecycle: active

Planning note: first v0.4.0 line after v0.3.37 release and DOI closeout.

## Purpose

v0.4.0 starts the first real GUI MVP line. The goal is not to replace the existing CLI or governance workflow. The GUI must expose the existing Python action layer safely and must not introduce a second command system.

## Boundary from v0.3.x

- v0.3.37 closes the pre-GUI hardening phase.
- No further general pre-GUI 0.3.x preparation loop is planned.
- Remaining test gaps are handled as entry conditions inside the first v0.4.0 GUI slices.
- v0.4.0 may start with contract and readiness slices before opening a real Tkinter window.

## Non-negotiable GUI principles

- The GUI uses the existing action registry and action specs.
- The GUI uses the existing mode guard.
- The GUI uses typed work orders or an equivalent typed execution contract.
- The GUI surfaces safety classes clearly.
- The GUI must preserve evidence and terminal-log discipline.
- The GUI must not invent its own release, merge, push, tag, or shell workflow.
- The GUI must not reintroduce removed shell adapters.

## MVP safety scope

- Read-only actions are allowed first.
- Bounded local actions are allowed only with explicit safety metadata and deterministic evidence behavior.
- Destructive actions are disabled in the first GUI MVP.
- Release, publish, merge, tag, delete, and remote-mutating actions are not directly executable from the first GUI MVP.
- Any future destructive GUI action requires a separate contract, tests, visible confirmation, and mode/evidence gates.

## First v0.4.0 slices

1. Strategy and roadmap update: define the v0.4.0 GUI line and lock the ordering.
2. GUI action readiness contract: prove that GUI-visible actions have names, descriptions, safety classes, expected outcomes, parameter rules, and evidence rules.
3. GUI dry-run entry point: provide a deterministic no-window launch check for Tkinter availability, action registry loading, safety policy loading, and mode guard availability.
4. Minimal Tkinter MVP: show action list, safety class, status, output, and a run path for allowed read-only or bounded actions.
5. GUI help baseline: add contextual help for actions and safety classes after the MVP execution path is stable.

## Test-gap handling

- Action registry and action specs are tested in the GUI action readiness slice.
- Typed work order execution as GUI input is tested before the first real run button is trusted.
- Evidence and dirty-log behavior are tested as action contracts, not as informal terminal habits.
- Tkinter is tested first through a dry-run or smoke path, not through fragile pixel tests.
- Controller and ViewModel tests are preferred over brittle GUI interaction tests.

## Later roadmap lines

- GUI help and guided mode.
- Installation assistant.
- Migration assistant.
- Didactic assistant.
- Documentation restructuring.
- Automatic website generation and maintenance.
- Public PyPI package maturity, after packaging, install, and support expectations are explicit.

## Explicit non-goals for the first GUI MVP

- No full release cockpit.
- No automatic merge or publish action.
- No installer.
- No migration assistant.
- No website generator.
- No second command registry.
