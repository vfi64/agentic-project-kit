# v0.4 Professional Single-User Strategy

## Purpose

This strategy paper defines a realistic path from the v0.3.12 state toward a professional single-user tool. It is a strategy document, not a release contract.

The goal is not to turn `agentic-project-kit` into an autonomous agent platform. The goal is to make it a reliable, understandable, and safe command-line companion for one maintainer working with AI assistants over long-running GitHub projects.

## Current baseline: v0.3.12

v0.3.12 is a strong foundation:

- deterministic governance is already solid (`doctor`, `check-docs`, `doc-mesh-audit`, release checks);
- the workflow cycle is bounded and reviewable;
- `./ns` and `./ns-menu` improve daily terminal usability;
- the read-only Pattern Advisor catalog introduces a first advisory layer without gates, automatic architecture decisions, workflow-state mutation, candidate capture, or promotion;
- releases, citation metadata, Zenodo verification, and handoff/state documents are part of the normal workflow.

External reviews of v0.3.12 are directionally useful, but they should not be treated as proof that the project is already finished. Grok correctly identifies the disciplined Pattern Advisor MVP and its boundary control. Gemini correctly emphasizes the didactic and process-control character of the project, but some of that didactic guidance predates v0.3.12 and should not be over-attributed to this release.

## Strategic diagnosis

The project is no longer an early MVP. It is already useful for a disciplined maintainer. The remaining gap is professional polish: installation, onboarding, command discoverability, recovery guidance, and user-facing documentation.

The core risk is not missing features. The core risk is uncontrolled feature expansion: adding `suggest`, migration, semantic checks, or automation before the daily user experience is clean enough to carry them.

Therefore, the next strategic priority is:

> Make the existing governed workflow easy to install, easy to understand, easy to recover, and hard to misuse.

## Professional single-user target

The project should be considered a professional single-user tool when one maintainer can do the following without deep repo knowledge:

1. Install the package cleanly.
2. Initialize or adopt a project in a guided way.
3. Understand the safe daily command path.
4. Run local gates and workflows without copy-and-paste fragility.
5. Recover from common failure states using clear instructions.
6. Read concise documentation that explains the tool identity, boundaries, and examples.
7. Trust that the tool will not silently mutate architecture, workflow state, or advisory decisions.

A realistic target for this is v0.5.0, not v0.4.0.

## Version strategy

### v0.3.13: tactical CLI and workflow usability cleanup

v0.3.13 should be a small tactical release. Its purpose is to reduce friction discovered during the v0.3.12 release cycle before starting a larger v0.4.0 milestone.

Recommended scope:

- add a documented TODO item or idea note for an `inspect` command;
- review the current public CLI surface and group it into daily, diagnostic, release, advisory, and maintenance layers;
- improve `./ns` and `./ns-menu` around release-related workflows only if this can be done without hidden automation;
- document where `./ns` ends and full `agentic-kit` commands begin;
- avoid adding Pattern Advisor `suggest`, migration, or semantic analysis in this release.

Definition of done:

- the maintainer can see the command layers at a glance;
- release and daily workflows are easier to choose;
- no new advisory automation is introduced.

### v0.4.0: Professional Polish milestone

v0.4.0 should be the first deliberately polished milestone. It should not be a feature explosion. It should focus on three pillars.

#### Pillar 1: Layered CLI and help system

Goal: make the command surface understandable.

Recommended work:

- define command layers: Daily, Diagnostic, Release, Advisory, and Maintenance;
- add a concise `agentic-kit help` or `agentic-kit assist` command only if it remains read-only;
- improve error messages with explicit next safe steps;
- add `--dry-run` where commands could otherwise mutate files, workflow state, or release state.

Non-goals:

- no hidden auto-fix behavior;
- no automatic command selection based on vague context;
- no advisory output becoming a deterministic gate.

#### Pillar 2: Installation and initialization experience

Goal: make the first ten minutes professional.

Recommended work:

- prepare for official PyPI distribution if the maintainer wants public installation;
- improve `agentic-kit init` so it explains profiles and policy packs in plain language;
- add a guided init path for common single-user cases: Python CLI, documentation-governed, research, and existing-repository adoption;
- add smoke tests for generated projects.

Non-goals:

- no full migration assistant in v0.4.0 unless it is read-only and explicitly scoped;
- no attempt to infer a complete architecture from a repository automatically.

#### Pillar 3: Documentation and public presentation

Goal: make the project understandable without reading the whole repo.

Recommended work:

- add a MkDocs-based documentation site or prepare the documentation structure for it;
- create a landing page explaining the problem: AI-assisted projects lose state unless the repository carries the memory;
- add tutorials for new project start, daily `ns` workflow, failed workflow recovery, release workflow, and Pattern Advisor read-only usage;
- add a small example project or generated fixture output.

Non-goals:

- no marketing claims that exceed actual behavior;
- no claim that semantic quality is proven by deterministic gates.

### v0.4.1 to v0.4.3: advisory and adoption expansion

Only after v0.4.0 professional polish should the project expand advisory intelligence.

Recommended order:

1. Pattern Advisor catalog growth while keeping entries read-only.
2. Pattern Advisor `suggest` prototype, read-only and explicitly advisory.
3. Existing repository adoption through `inspect` or `migrate --dry-run`, producing reports before changes.
4. Profile and template expansion.

### v0.5.0: Production-ready single-user

v0.5.0 should be the point where the tool feels complete for one serious maintainer:

- installation is straightforward;
- init and adoption workflows are guided;
- command layers are documented and discoverable;
- failure recovery is clear;
- a documentation site exists or the docs are structured as if they could be published immediately;
- Pattern Advisor remains useful but advisory;
- at least one real larger project can be bootstrapped or managed with the kit without chat-history dependence.

## Prioritized gap list

### Must-have

1. Better CLI discoverability and layering.
2. Better onboarding and guided init.
3. Better failure messages and recovery paths.
4. Official distribution path or explicit decision to postpone PyPI.
5. Professional documentation entry point.

### Strongly recommended

1. Read-only `inspect` for repository state and adoption readiness.
2. Standardized examples and generated-project fixtures.
3. More profiles and policy-pack explanations.
4. Pattern catalog growth after CLI polish.
5. Clear configuration strategy for user preferences.

### Nice-to-have

1. Pattern `suggest` as read-only advisory command.
2. Migration assistant, initially report-only.
3. Semantic quality advisory checks.
4. LLM-switch test helpers.
5. Custom templates and themes.

## Architectural guardrails

The following rules should remain stable through v0.4 and v0.5:

- deterministic gates check structure, drift, release state, and known bad patterns; they do not prove semantic excellence;
- advisory systems must remain advisory unless a separate deterministic gate is explicitly designed and documented;
- commands that mutate state must say so clearly;
- read-only commands must stay read-only;
- one semantic slice per PR;
- documentation must be updated in the same slice as public behavior;
- no hidden autopilot;
- no automatic architecture choice;
- no workflow-state mutation from advisory commands.

## Recommended next move

Do not start v0.4.0 immediately with MkDocs, PyPI, Pattern Advisor expansion, and init redesign all at once. That would be too broad.

Start with v0.3.13 as a short stabilizing strategy-to-implementation bridge:

1. inventory the command surface;
2. define command layers;
3. add or specify `inspect` as a read-only diagnostic/adoption command;
4. decide whether `./ns release` should exist and exactly what it may do;
5. update the handoff and status documents with the v0.4.0 pillar plan.

Then cut v0.3.13 if the CLI strategy is clearer. After that, begin v0.4.0 with help/layering work first, not with Pattern Advisor automation.
