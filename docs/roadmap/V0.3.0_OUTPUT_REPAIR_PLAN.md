# v0.3.0 Output Repair Plan

Status: draft planning note
Baseline: after v0.2.11

## Goal

v0.3.0 should add a bounded output-repair pipeline on top of the existing output-contract validation and machine-readable validation reports.

The goal is not to let an LLM freely rewrite outputs. The goal is to make missing or malformed required sections detectable, reportable, and repairable under explicit constraints.

## Existing foundation

- `validate-output-contract` validates generated text against output contracts.
- `--report` writes a machine-readable JSON validation report.
- `--report-schema` validates the generated report shape before writing it.
- governance-wrapper projects generate `docs/output-contracts/default-answer.yaml` and `docs/schemas/validation-report.schema.json`.
- `CURRENT_WORKFLOW_OUTPUT.md` supports app-based workflows without accumulating raw historical dumps.

## Non-goals

- Do not add broad LLM-as-judge scoring in the first v0.3.0 slice.
- Do not let repair rewrite otherwise valid final content.
- Do not hide the original output.
- Do not make repair implicit or silent in CI.
- Do not mix repair policy with project-specific Comm-SCI-Control semantics in the generic kit.

## Proposed architecture

1. Parser
   - Parse required section headers and section bodies from plain-text or Markdown output.
   - Preserve original text offsets where practical.

2. Validator
   - Reuse output-contract validation as the source of truth.
   - Keep findings machine-readable and schema-validatable.

3. Repair planner
   - Convert findings into explicit repair operations.
   - Initial operations should be small and deterministic, for example inserting missing required section headings with placeholder-free user-visible failure text only when allowed by policy.

4. Repair applier
   - Apply only allowed operations.
   - Preserve original output and emit repaired output separately.

5. Evidence report
   - Emit a machine-readable repair report.
   - Include original validation findings, applied operations, skipped operations, and final validation status.

## Suggested CLI shape

```bash
agentic-kit repair-output-contract output.md \
  --contract docs/output-contracts/default-answer.yaml \
  --out repaired-output.md \
  --report repair-report.json \
  --report-schema docs/schemas/repair-report.schema.json
```

The command should fail if the output cannot be repaired under the configured policy.

## First implementation slice

1. Define a repair-report JSON shape and generated schema.
2. Add tests for missing required section detection.
3. Add a deterministic repair planner that can classify repairable vs non-repairable findings.
4. Add a dry-run command or flag before writing repaired output.
5. Add E2E coverage in a generated governance-wrapper demo.

## Application to Comm-SCI-Control

Comm-SCI-Control should not directly inherit generic repair semantics. Instead, the generic kit should provide parser/validator/report/repair primitives that Comm-SCI-Control can map to its stricter JSON rule set, SCI variants, audit stream, and bounded delta-repair requirements.

## Open decisions

- Should v0.3.0 repair only section presence, or also section ordering?
- Should placeholder insertion ever be allowed, or should missing sections remain unrepaired unless content is supplied by a bounded model call?
- Should LLM-assisted repair be out of scope until a later v0.3.x release?
- Should repair reports reuse the validation-report schema style or use a separate schema from the beginning?
