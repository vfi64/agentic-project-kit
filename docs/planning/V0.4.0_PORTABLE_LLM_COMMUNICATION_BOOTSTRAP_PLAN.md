## Problem statement
The project has strong governance rules, but the latest workflow exposed a critical weak point: chat-generated terminal blocks still assumed a POSIX-like shell toolchain and a healthy `PATH`. The block failed before it could even preserve evidence because tools such as `git`, `cp`, `tail`, and `sh` were not found. This is not only a terminal issue; it is a communication-contract issue.
The kit must no longer rely on long shell blocks as canonical workflow execution. Large LLMs must be constrained by repository-owned, portable, tested contracts and runners. The shell may remain a convenience adapter; durable behavior belongs in Python modules, CLI commands, tests, docs, and machine-readable context.
## Non-negotiable objectives
1. Successor chats must fetch and read the current mandatory rule sources before they produce mutation instructions.
2. Communication rules, final summaries, evidence rules, drift detection, and handoff-prompt generation must be repository-owned and test-backed.
3. Canonical execution must be operating-system independent: Python core first, no POSIX-only tool assumptions in critical workflow control.
4. Drift must be detected deterministically and must stop mutation-oriented work until a warning and handoff prompt are produced.
5. No log upload or recovery step may relabel failed work as successful.
6. No future workflow may use `REMOTE_EVIDENCE: PENDING` as a final summary value.
7. New blocks must not append handwritten legacy footers after the summary renderer.
8. Every implementation step must explicitly look for a better path toward determinism, portability, consistency, and lower LLM discretion.
## Required source-of-truth hierarchy for successor chats
A successor chat must begin by reading these sources, in this order, before proposing any terminal block or mutation:
1. `.agentic/compiled_agent_context.yaml`
2. `docs/governance/FINAL_SUMMARY_CONTRACT.md`
3. `docs/governance/CHAT_COMMUNICATION_CONTRACT.md`
4. `docs/governance/PORTABLE_CHAT_EXECUTION_CONTRACT.md`
5. `docs/governance/CHAT_BOOTSTRAP_AND_DRIFT_CONTRACT.md`
6. `docs/TEST_GATES.md`
7. `docs/STATUS.md`
8. `docs/handoff/CURRENT_HANDOFF.md`
9. relevant source files and tests for the requested slice
If any required source is missing or contradictory, the successor chat must report drift, avoid mutation, and offer a comprehensive handoff prompt.

Add or sharpen these documents:

- `docs/governance/PORTABLE_CHAT_EXECUTION_CONTRACT.md`
- `docs/governance/CHAT_BOOTSTRAP_AND_DRIFT_CONTRACT.md`
Update:

- `docs/TEST_GATES.md`
- `docs/STATUS.md`
- `docs/handoff/CURRENT_HANDOFF.md`
- `.agentic/compiled_agent_context.yaml`
- `.agentic/handoff_state.yaml`
- `docs/DOCUMENTATION_COVERAGE.yaml`
Optimization check for Phase 1: prefer one canonical rule per concept and cross-reference it instead of duplicating paragraphs across status, handoff, and roadmap files.
### Phase 2: portable Python checks
Implement Python-first checks:
- `agentic-kit bootstrap-check`
- `agentic-kit comm-rules-check`
- `agentic-kit handoff-prompt --reason drift`
These checks must use `pathlib`, `platform`, `shutil.which`, and `subprocess.run([...], shell=False)` only where subprocesses are unavoidable. They must not require POSIX tools such as `cp`, `tail`, `grep`, `sed`, `head`, or `sh`.
Optimization check for Phase 2: if a workflow can be verified by reading files or importing constants, prefer that over running external commands.
### Phase 3: summary renderer alignment
Align `src/agentic_project_kit/run_summary_renderer.py` with `docs/governance/FINAL_SUMMARY_CONTRACT.md`:
- support `NO-COMMAND` where the contract declares it,
- restrict `REMOTE_EVIDENCE` to `PASS`, `FAIL`, `PARTIAL`, `NOT_REQUIRED`,
- reject `PENDING`, `NONE`, and `CHAT_ONLY` for `REMOTE_EVIDENCE`,
- produce precise validation errors,
- require honest remote evidence paths when `REMOTE_EVIDENCE: PASS` is claimed.
Optimization check for Phase 3: avoid parallel summary schemas; renderer, validator, docs, and tests must share one vocabulary.
### Phase 4: deterministic drift detection
The communication-rules check must detect at least:
- contract-vs-renderer value drift,
- contract-vs-test drift,
- missing bootstrap source references,
- missing communication-contract anchors in status and handoff,
- forbidden final summary values,
- shell-only canonical workflow examples,
- handoff prompt omissions.
Optimization check for Phase 4: fail closed. If the checker cannot decide whether a required anchor exists, it must report drift rather than pass.
### Phase 5: handoff prompt generation
The handoff prompt generator must include:
- repo and remote identity,
- branch and commit if available,
- required first-read source list,
- active summary and communication contracts,
- current drift findings,
- forbidden patterns,
- last safe state,
- next safe step,
- instruction not to mutate until sources are read.
Optimization check for Phase 5: make the generated prompt short enough for practical use but complete enough that a new chat can continue without memory.
## Acceptance criteria
This work is not complete until:
- the plan and roadmap name portable LLM communication hardening as the current top priority,
- the communication rules are separate and explicit,
- portable execution is separate and explicit,
- bootstrap and drift behavior are separate and explicit,
- summary renderer, docs, and tests are aligned,
- `check-docs` protects the new anchors,
- a deterministic check detects drift,
- a drift handoff prompt path exists,
- no canonical workflow depends on POSIX shell utilities for correctness.
## Current next safe step
Create the documentation contracts first, then add tests and Python checks. Do not continue GUI feature work until communication, summary, and bootstrap drift hardening is at least documented and test-gated.
